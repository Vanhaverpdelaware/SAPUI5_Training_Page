Strict Mode

TypeScript strict mode catches a class of bugs at compile time that would only surface at runtime in JavaScript. At delaware, strict: true is mandatory (ADR-002).

What strict mode enables

Flag What it prevents
strictNullChecks null/undefined bugs — must handle them explicitly
noImplicitAny Implicit any types — forces explicit typing
noImplicitReturns Functions that don't always return a value
noUnusedLocals Unused local variables (dead code)
noUnusedParameters Unused function parameters

Common strict mode patterns in UI5

Handling optional chaining (?.getView())

With strictNullChecks, getView() can return undefined. Use optional chaining:

// ❌ Fails strict check
const oModel = this.getView().getModel("view") as JSONModel;

// ✅ Correct with strict mode
const oModel = this.getView()?.getModel("view") as JSONModel;

Non-null assertion for framework-initialized properties

UI5 sets some properties via the framework after the constructor. Use ! to tell TypeScript these will be set before use:

// The framework initializes this before onInit() is called
private _oViewModel!: JSONModel;
private _oDataModel!: ODataModel;

Type assertions for event parameters

public onInputChange(oEvent: Event): void {
  // getParameter() returns unknown — cast it
  const sValue = oEvent.getParameter("value") as string;
  const nValue = oEvent.getParameter("selectedIndex") as number;
}

ESLint rules at delaware

From TDD-001, the project ESLint config enforces:

{
  "rules": {
    "@typescript-eslint/no-explicit-any": "error",
    "@typescript-eslint/explicit-function-return-type": "warn",
    "@typescript-eslint/no-unused-vars": ["error", {
      "argsIgnorePattern": "^_"
    }],
    "prefer-const": "error",
    "no-var": "error"
  }
}
Tip: Prefix intentionally unused parameters with _ to satisfy the argsIgnorePattern rule — e.g. _oEvent for an event parameter you receive but don't use.
