MVC Pattern

SAPUI5 follows the Model-View-Controller (MVC) architectural pattern. Every UI5 application separates concerns into three distinct layers.

The Three Layers

Layer Responsibility File type
Model Holds data (JSON, OData V4, i18n) .ts / manifest.json
View Declares the UI structure declaratively .view.xml
Controller Handles user interaction and business logic .controller.ts

Project Folder Structure

A standard delaware UI5 project follows this structure inside webapp/:

webapp/
├── Component.ts           # App entry point
├── manifest.json          # App descriptor
├── index.html             # Bootstrap
├── controller/
│   ├── App.controller.ts
│   └── List.controller.ts
├── view/
│   ├── App.view.xml
│   └── List.view.xml
├── model/
│   ├── formatter.ts
│   └── types.ts
└── i18n/
    └── i18n.properties

Why MVC?

delaware standard: Avoid putting all logic in controllers. Extract reusable logic into TypeScript classes in the model/ folder. Large controllers become hard to maintain and cause duplicate code. See the Class-based development guide on Confluence.

Separating concerns means:

  • Views can be redesigned without touching business logic
  • Controllers stay thin — they orchestrate, not implement
  • Models are reusable across multiple views
